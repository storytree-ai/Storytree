# Gas Town & loom vs the three surfaces

storytree frames its design around three "attention surfaces" that give an agent a map of the work: a **library** (a curated, validated knowledge tier), the **storytree** itself (the codebase as an authored top-down DAG of stories, capabilities, and contracts under explicit proof modes), and a **noticeboard** (a dependency-aware task tier that acts as the agent's external memory). Assessed against this triad, two of the most visible 2026 agent-orchestration projects — Steve Yegge's [Gas Town](https://github.com/steveyegge/gastown) and Geoffrey Huntley's [loom](https://github.com/ghuntley/loom) — converge strongly on roughly **one** of the three surfaces, the noticeboard, and on neither of the other two. Gas Town's Beads substrate is a strong, direct analog of a dependency-aware task tier; loom's task/code/knowledge surfaces are unintegrated infrastructure rather than agent-facing maps. Neither project frames its design as a deliberate set of three surfaces, and neither independently arrives at storytree's curated library or its authored code-graph/DAG. The convergence is therefore real but partial and asymmetric: it corroborates the noticeboard idea, not the unified triad.

## Gas Town (Steve Yegge)

**What it is.** Gas Town is a multi-agent orchestration system for coordinating many AI coding agents (Claude Code, GitHub Copilot, Codex, Gemini) with persistent work tracking. Its README opens by describing it as a "Multi-agent orchestration system for Claude Code, GitHub Copilot, and other AI agents with persistent work tracking," and Yegge frames it as "a new take on the IDE for 2026." The repository is predominantly Go. Yegge announced the launch in early January 2026 ("launched Gas Town!" in the [Welcome to Gas Town](https://steve-yegge.medium.com/welcome-to-gas-town-4f25ee16dd04) post, with the [Future of Coding Agents](https://steve-yegge.medium.com/the-future-of-coding-agents-e9451a84207c) article published days prior).

**Architecture (Town > Rigs > Polecats > Mayor).** Gas Town is organized as a hierarchy: a **Town** contains **Rigs** (git-repo containers), which host **Polecats** (worker agents), all coordinated by a **Mayor** AI. Work is persisted via git. Agent context and attention are managed not through a knowledge map but through persistence and coordination *infrastructure*: git-worktree Hooks, a **Seance** session-recovery mechanism that "discovers previous agent sessions via `.events.jsonl` logs," and mailboxes/handoffs between agents. The emphasis is on decomposing work into micro-workflows and farming it out across an "agent village," addressing the fact that agent sessions are short and memoryless.

**Beads / the MEOW stack / Convoys — a strong noticeboard analog.** The substrate beneath Gas Town is [Beads](https://github.com/steveyegge/beads), Yegge's standalone issue-tracker / external-memory system, [introduced here](https://steve-yegge.medium.com/introducing-beads-a-coding-agent-memory-system-637d7d92514a). Beads provides "persistent, structured memory for coding agents. It replaces messy markdown plans with a dependency-aware graph." It is purpose-built for the reality that agents "have no memory between sessions — sessions that only last about ten minutes," replacing flat markdown TODOs with a queryable, dependency-aware graph (parent/child plus blocking pointers across four link types). It auto-computes ready work: `bd ready` lists "tasks with no open blockers." Above Beads, Gas Town bundles units of work into **Convoys** — per the README, "Work tracking units. Bundle multiple beads," which Yegge describes as "Gas Town's ticketing or work-order system... Convoys are basically features." The broader [Gas City](https://steve-yegge.medium.com/welcome-to-gas-city-57f564bb3607) writeup describes a **MEOW stack** (Beads > Epics > Molecules > Protomolecules > Formulas > Wisps) that "places Work front and center, as the first-class system primitive... creating a versioned knowledge graph of all your issues and tasks." This dependency-aware, versioned, git-backed graph of issues and tasks is a strong, direct analog of storytree's noticeboard.

**The absence of a library or code-graph surface.** Term-by-term searches of Gas Town's primary sources found none of: documentation tier, wiki, library, knowledge base, code graph, code map, AST, embeddings, RAG, or semantic search. Gas Town's surface is task decomposition and coordination, not a curated knowledge tier and not an authored or derived map of the codebase. The one apparent near-miss — secondary descriptions of Beads as a "knowledge graph" — refers to a graph *of tasks*, which maps onto the noticeboard, not onto a library or a code-graph. Gas Town therefore converges on the noticeboard surface and on neither of the others.

## loom (Geoffrey Huntley)

**What it is.** [loom](https://github.com/ghuntley/loom) is an AI-powered coding agent written in Rust — a REPL/terminal interface to LLM agents that execute tools. Its README states: "Loom is an AI-powered coding agent built in Rust... REPL interface... three core principles: Modularity, Extensibility, Reliability." It is organized around a large, modular spec set (on the order of 50–60 spec files across 30+ crates), structured by *functional subsystem* rather than by attention surface.

**Core architecture (state machine, tool registry, LLM proxy, Kubernetes "Weaver").** loom's key components are a Core Agent built as a **state machine**, a **Tool System** (a registry and execution layer), a server-side **LLM Proxy**, and **Weaver**, a remote-execution layer running on Kubernetes pods. The [spec index](https://github.com/ghuntley/loom/blob/trunk/specs/README.md) is organized by LLM integration, observability, security/identity (ABAC/SCIM/audit), and remote execution — these are infrastructure subsystems, not the three surfaces storytree describes.

**FTS5-over-threads + web search as its only retrieval.** loom's knowledge-retrieval surface is narrow: an FTS5 full-text search over conversation threads (the spec's `search-system.md` is "FTS5 search for threads"; the **Thread System** handles "Thread persistence and sync") plus external web search via Google Custom Search (`web-search-system.md` = "Google CSE integration"). There is no schema-validated knowledge/library tier. A `docs-system.md` (Diátaxis/MDX/Pagefind) exists, but it is a human-facing documentation *publisher*, not an agent-facing knowledge tier.

**Plain git/jj VCS tooling as its code surface.** loom's code surface is conventional version-control tooling — an `scm-system`, a jj-based "spool" system, `git-metadata` ("Git context tracking in threads"), auto-commit, and a GitHub App — rather than a semantic code-graph or an authored DAG of stories, capabilities, and contracts. Huntley's own teaching (his [six-month recap](https://ghuntley.com/six-month-recap/) and the [loop](https://ghuntley.com/loop/) writeup) emphasizes spec-driven workflows, headless agents cloning software (Tailscale, HashiCorp Nomad, Infisical), and transient context — using the context window for one activity and then *clearing* it. That is closer to the opposite of a persistent map. The nearest parallel to storytree's library is Huntley's practice of "maintaining a standard library of prompts," but this is a tactical habit with no database, schema, versioning, or queryable tier — not an architected knowledge surface.

**The absence of a task board, code-graph, or validated knowledge tier.** The README's component table (roughly eight rows) lists no task-board/kanban/noticeboard surface, no semantic code-graph, and no validated knowledge/library tier. loom thus has the *components* one might assemble into surfaces (VCS tooling, thread search, docs) but none of them is presented as an integrated, agent-facing attention surface — and unlike Gas Town it has no dependency-aware task tier at all.

## Scorecard

Legend: ✅ strong, direct analog · ⚠️ weak, partial, or unintegrated parallel · ❌ absent.

| Project | Library (curated/validated knowledge tier) | storytree / code-graph (authored DAG: story > capability > contract) | Noticeboard (dependency-aware task tier) |
| --- | --- | --- | --- |
| **Gas Town** (Yegge) | ❌ No knowledge tier; "knowledge graph" framing refers to a task graph | ❌ Git-backed work persistence and coordination infra, not a code-graph or authored DAG | ✅ Beads + MEOW stack (Convoys/Epics/Molecules): versioned, dependency-aware, git-backed task graph with auto-computed ready work |
| **loom** (Huntley) | ⚠️ FTS5 thread search + web search + ad-hoc "standard library of prompts" — retrieval/habit, not a validated tier | ❌ Plain git/jj VCS tooling; teaching favors transient/cleared context over a persistent map | ⚠️ No task board; thread persistence is conversation memory, not a dependency-aware work tier |
| **storytree** | ✅ Curated, validated knowledge tier | ✅ Codebase as an authored top-down DAG under explicit proof modes | ✅ Dependency-aware task/work tier as agent external memory |

## Caveats

- **Time-sensitivity (Jan–Jun 2026 snapshot).** All sources date from January to June 2026, and both projects move quickly. Beads' storage evolved from git-backed JSONL to a Dolt SQL backend between Yegge's announcement article and the current repository state; Gas Town extended into a "Gas City" form by April 2026; loom is on an actively developed trunk. Specific architectural details may have shifted since.
- **The noticeboard and library mappings are analytical framing, not the authors' own words.** Neither Yegge nor Huntley describes their work in terms of storytree's "three surfaces" or a deliberate "map" for agent attention. Casting Beads as a noticeboard analog, and Huntley's prompt habit as a library parallel, is this assessment's analogical bridge, not a claim made by the projects themselves. The prompts-as-library parallel in particular is weak and hedged.
- **The unverified loom OracleTool lead.** loom's deeper spec set was reported to contain an `OracleTool` / server-side knowledge-base component that could loosely map to a library function. This lead was not verified against primary sources and is not framed by loom as an attention surface; whether it constitutes a real curated/validated knowledge tier remains an open question.
- **Convergence is partial and possibly entangled.** The strongest, most independent corroboration is for the noticeboard surface (Yegge/Beads). Whether storytree's explicit three-surface framing is genuinely novel — or simply names and unifies primitives (task tracker + docs + VCS) that Gas Town and loom already have in unintegrated form — and whether these are independent inventions or share lineage through Yegge's "external memory" framing and Huntley's spec-driven practices, are open questions this snapshot does not resolve.
