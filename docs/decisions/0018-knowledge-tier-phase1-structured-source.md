# ADR-0018: Knowledge tier Phase 1 ‚Äî structured source of truth, generated views, corpus curation

## Status

accepted (2026-06-08) ‚Äî **resolves** [ADR-0017](0017-cross-cutting-knowledge-tier.md)'s deferred
"templates ‚Üí schema"; builds on [ADR-0013](0013-structured-corpus-markdown-as-view.md) (structured
corpus), [ADR-0016](0016-knowledge-code-binding-and-staleness.md) (binding/staleness), and
[ADR-0017](0017-cross-cutting-knowledge-tier.md) (the knowledge tier); applies
[ADR-0011](0011-own-the-agent-loop-and-context-engineering.md)'s pi‚Üíowned-loop sweep to the corpus.
Records owner decisions taken through `open-question` artifacts in the studio.

## Date

2026-06-08

## Context

The cross-cutting knowledge "Library" (ADR-0017) existed as a git stopgap: `apps/studio/data/assets.json`
was the runtime store, regenerated by `seed.assets.mjs`, which auto-extracted definitions from
`docs/glossary.md` (one paragraph per term) and hard-coded the rest. Two sources competed; a `--force`
reseed reverted hand edits; and the multi-section templates could not live in the single-paragraph
glossary. The owner directed a proper rebuild and to drive decisions **through the Library UI** rather
than chat. Each decision was parked as an `open-question` unit, commented on in the studio, and is
recorded here as it resolves.

## Decision

1. **Structured source of truth.** `apps/studio/data/knowledge.json` is the single source ‚Äî typed units
   validated by a `packages/core` zod `Knowledge` union discriminated by `kind`. `assets.json` (the
   studio view) and `docs/glossary.md` (the glossary view) are **generated** from it by
   `build-corpus.mjs`. The old `seed.assets.mjs` and glossary-as-source are **retired**. *(Resolves the
   durability question, ex-`oq-library-body-durability`: the store is the source; there is no competing
   generator left to revert it.)*

2. **Templates are schema-derived.** One ordered `KIND_SPECS` table per kind drives the zod schema, the
   body renderer, and the blank-template generator ‚Äî one table, three consumers. This **resolves
   ADR-0017's deferred "templates ‚Üí schema"**: a guardrail's "Enforced by" is now a required field
   (not a save-time check), and templates cannot drift from the schema.

3. **Glossary is a generated view.** Each term's canonical blurb lives in its structured unit
   (`glossaryBody` / `glossaryTerm` / `glossarySection`); `docs/glossary.md` is regenerated from them
   (zero-loss, byte-identical at cutover). `glossary-wins` still holds ‚Äî the glossary is now the
   *rendered* canonical view, edited via its source.

4. **Vocabulary reconciled to the owned loop.** The corpus was swept pi ‚Üí owned-loop / `packages/agent`
   to match the glossary and ADR-0011 (its tracked follow-up sweep).

5. **Curation decisions** (owner, via studio open-questions):
   - **Redundant pairs merged & retired** (ex-`oq-redundant-library-pairs`): `own-the-layers` absorbs
     the v1-scar evidence of the retired `vibe-the-load-bearing-layers`; the guardrail
     `human-owns-the-outer-loop` absorbs the retired `inner-loop-outer-loop`'s glossary term.
   - **Thin terms consolidated** (ex-`oq-thin-glossary-terms`): the six lifecycle statuses ‚Üí one
     `lifecycle-status` definition; `outcome` / `guidance` / `title` / `id` ‚Üí one `unit-fields`
     definition (every member preserved inside).
   - **Recategorizations** (ex-`oq-soft-recategorizations`): `defects-amend-the-owning-story` and
     `fail-closed-on-dirty-tree` ‚Üí `principle` ("less is more").

6. **The open-question lifecycle (the operating workflow).** Owner decisions are parked as
   `open-question` units; the owner comments; when an agent judges the feedback **resolves** the
   question, it records the decision in an ADR (this one) and **retires** the open-question ‚Äî no manual
   close by the owner. Deferred feedback leaves the question open. This ADR is the first application.

## Consequences

- Library units **94 ‚Üí 82** (curation + retiring the four resolved open-questions); `assets.json` and
  `docs/glossary.md` are now pure derivatives of `knowledge.json`.
- The structured source is the input to **Phase 2** ‚Äî migrating the corpus into the provisioned Cloud
  SQL store ([ADR-0015](0015-gcp-hosting-cloud-sql-event-store.md) / ADR-0017).
- **Known tradeoffs (for the DB phase):** `glossaryBody` duplicates content with the Library body
  fields (two renderings of one unit ‚Äî dedupe in the store); three ids lag their titles
  (`pi-adapter` / `pi-event-stream` / `stack-pi-coding-agent`); the code package on disk is still
  `packages/pi-adapter` while the corpus uses the glossary's `packages/agent` term.

## What this does NOT decide

- **The tier's name** (`oq-knowledge-tier-name`, open) and the **store execution model ‚Äî DBOS vs a
  plain Postgres connection** (`oq-store-execution-model`, open) ‚Äî both parked for Phase 2.
- **Cautionary-pattern housing** (`oq-anti-pattern-lessons`) ‚Äî deferred by the owner. *(Resolved
  later by [ADR-0028](0028-merge-cautionary-lessons-into-counterparts.md): merge & retire ‚Äî each lesson's
  v1 scar folded into its counterpart's "Why".)*
- The full **agent‚ÜîLibrary interaction protocol** (read / edit / curate; the resolve‚Üíretire automation)
  ‚Äî under design separately.
